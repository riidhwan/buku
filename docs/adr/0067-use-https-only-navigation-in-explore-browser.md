# Use HTTPS-only navigation in Explore Browser

The Explore Browser never loads cleartext HTTP traffic. Every top-level GET navigation to an HTTP target is silently replaced by an Explore HTTPS Upgrade that preserves the rest of the URL, except that explicit HTTP port 80 maps to HTTPS port 443 while other explicit ports remain unchanged. The rule applies regardless of whether the target came from address entry, a page, Reading Mode or new-window link, Chapter Navigation, tab restoration, or back navigation; new-window targets continue in the current Explore Browser Tab. Successful navigation persists only the final committed HTTPS URL; mixed HTTP subresources and non-GET HTTP navigations are blocked rather than rewritten.

When an upgraded target cannot establish a secure connection, has an invalid certificate, redirects back to HTTP, repeats an HTTP target, or exceeds ten upgrades in one navigation chain, Buku fails closed with a typed, reason-specific failure state. Offline failures use ordinary network messaging and do not promote an insecure handoff; ambiguous online DNS, timeout, or connection failures say only that the secure site could not be reached rather than claiming the site is HTTP-only. Retry attempts HTTPS only. Where the original action can be represented safely, the user may explicitly hand its URL to the device browser; Buku does not hand off insecure form submissions whose request body it cannot preserve.

The failure state is rendered by Ionic beneath the retained Explore Browser chrome while the native WebView is hidden; Buku does not inject an error document into the WebView. This keeps failure reasons and recovery actions inside the application workflow and preserves a testable presentation boundary.

While failure is visible, browser Back or Android Back dismisses it and restores the last committed page without consuming the Explore Browser Back Stack; a tab with no committed page returns to blank. Failure state is transient and discarded by tab switching, tab closing, or closing Explore Browser, consistent with Explore Browser Tabs remaining URL slots rather than restored live sessions.

This accepts incompatibility with HTTP-only sites and potentially incomplete pages that depend on mixed content in exchange for keeping Explore discovery traffic protected. Buku navigates directly without a separate availability probe and never enables global Android cleartext traffic; the device browser owns any warning or exception flow for externally opened targets.
